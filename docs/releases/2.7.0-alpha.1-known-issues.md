# 2.7.0-alpha.1 Known Issues and Validation Gaps

## Release blockers

No automated or local-browser release blocker is currently recorded. This status is conditional on the final clean verification gate passing.

Production remains blocked until physical-device RC validation is complete.

### RC Hotfix 1 — missing Contract Due Date on physical device

- Reported result: the Contract Due Date controls were not found during the physical RC pass.
- Source evidence: the RC contained one active control and no hiding selector; automated rendered checks also found non-zero visible controls. The prior test only asserted DOM presence, not viewport visibility or explicit grouping.
- Fix: the active controls are now an explicitly labelled form group with a distinct visual marker, covered at all required RC viewports, and the app-shell cache advances to `v2.7.0-alpha.17`.
- Status: fixed automatically; **physical retest NOT TESTED** and still release-blocking for production.

### RC Hotfix 2 — Contract Due Date placement

- Request: place the existing shared control immediately below the relevant mode-specific Next Appointment Date.
- Implementation: one existing DOM group is relocated by the presentation layer between the in-person EOI and Zoom Client Review anchors; it is never duplicated.
- Conditional handling: when optional EOI details are hidden, the group remains visible immediately after the hidden EOI content container.
- Status: implemented with cache `v2.7.0-alpha.18`; **physical placement confirmation NOT TESTED**.

## Non-blocking known issues

### RC Hotfix 5: native dates and two-file package download

- Reported result: the visible date inputs did not match adjacent controls on physical devices, and Share Package was unreliable for the PDF-plus-ZIP handover.
- Implementation: the three active date inputs now share a narrow native-input style; Download Package initiates the current Combined PDF and Document ZIP and then directs staff to Prepare Email.
- Browser limitation: a browser can suppress the second automatic download and does not expose reliable file-write completion. The UI reports only that initiation started and keeps Save Combined PDF and Save ZIP as explicit fallbacks.
- Status: implemented with cache `v2.7.0-alpha.21`; **physical date-picker, Safari two-download, Files, and manual email-attachment retest NOT TESTED**.

### RC hotfix — Zoom timeline and whiteboard

- Reported result: the Zoom timeline rendered as eight vertical full-width rows and the revealed whiteboard was not drawable.
- Root causes: the generic Zoom visibility rule overrode the timeline's flex display, while the canvas was sized to zero during hidden initialization and was not resized on reveal.
- Status: fixed automatically with a Zoom-timeline-specific display rule and a whiteboard-container resize observer in cache `v2.7.0-alpha.20`; **physical retest NOT TESTED**.

### RC Hotfix 3 — Sale Details finance layout

- Previous concern: in-person Next Appointment and Contract Due Date controls left an awkward layout gap, while a single finance percentage could not represent participating clients independently.
- Implementation: in-person finance is split by client and the EOI date/time group sits beside the unchanged Contract Due Date group; Zoom retains its existing Client Review schedule without finance controls or an invented time field.
- PDF limitation: the standard EOI contains one verified finance-percentage location and receives Client 1. No approved separate Client 2 target exists in the standard or La Vida templates, so no PDF geometry or overlay coordinates were guessed.
- Status: implemented with cache `v2.7.0-alpha.19`; **physical layout and finance-control retest NOT TESTED**.

- LAN HTTP is intentionally used for local device layout/workflow testing. On iPhone it is not a trusted secure context, so it cannot validate service-worker installation or native Web Share capability.
